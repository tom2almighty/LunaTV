# LunaTV 后端 API 统一化与前端流畅度改造文档

## 1. 背景

本次审查目标是两件事：

1. 后端 API 端点统一度（鉴权、错误结构、响应结构、测试覆盖）。
2. 前端页面流畅度（搜索页与首页为重点）。

## 2. 当前现状（代码基线）

- API 路由共 `41` 个，已使用统一 handler 的 `30` 个，未统一的 `11` 个。
- API 返回结构同时存在：`{ ok: true }`、`{ success: true }`、`{ error: string }`、业务对象裸返回。
- 鉴权路径存在分叉：部分接口仅检查 cookie 是否有 `username`，未统一走 `requireActiveUsername`。
- 前端存在持续 `requestAnimationFrame` 轮询检测滚动，属于常驻主线程任务。
- 搜索页虚拟网格在结果流式增长期间会重复重建布局监听，造成无效开销。

## 3. 主要问题

### P0：鉴权一致性与可信度

- 部分 RAW 路由绕过统一鉴权，无法自动复用「用户存在/封禁」校验。
- 风险：不同接口对同一用户状态的判断不一致。

### P1：响应协议不统一

- 客户端需要分支兼容多种返回形态，增加维护成本和回归风险。

### P1：前端滚动检测策略低效

- back-to-top 可见性采用常驻 raf 轮询，页面空闲时仍占用计算。

### P2：搜索页渲染有额外 CPU 开销

- 聚合统计重复计算。
- 虚拟网格监听器重建频率高于必要值。

### P2：调用层未收敛

- 仍有大量页面直接 `fetch('/api/...')`，错误处理与登录过期处理逻辑分散。

## 4. 改造目标

- 所有业务 API 统一进入 `executeApiHandler / executeAdminApiHandler`。
- API 协议统一为：
  - 成功：`{ success: true, data: ... }`（下载流/二进制类接口除外）
  - 失败：`{ success: false, error: { code, message, details? } }`
- 前端滚动与虚拟列表逻辑无常驻轮询，减少无效重绘。
- 搜索页聚合统计降重计算。
- 关键变更有测试覆盖，避免回归。

## 5. 范围

### In Scope

- `src/app/api/**` 中 RAW 业务路由统一化。
- `src/server/api/**` 扩展通用校验与错误映射。
- 搜索页相关 hooks（滚动可见性、虚拟网格、聚合统计）。
- 前端高频调用处收敛至统一 API 客户端。

### Out of Scope

- UI 视觉改版。
- 数据库 schema 调整。
- 全量页面重写。

## 6. 分阶段实施策略

1. **后端契约与鉴权层先行**：先统一协议与鉴权，再改调用端，避免前后端并发改动导致调试困难。
2. **前端性能热点收敛**：先处理常驻轮询和虚拟网格重建，再做细粒度优化。
3. **调用层逐步迁移**：高风险页面（登录、管理、用户设置）优先迁移，保留短期兼容。

## 7. 验收标准

- RAW 业务路由数减少到可接受最小集合（仅公共只读/下载型例外）。
- 关键接口在错误与成功场景返回结构可预测。
- 搜索页与首页交互中无持续轮询逻辑。
- 新增/修改测试全部通过：`pnpm test:run`、`pnpm lint`、`pnpm typecheck`。

## 8. 风险与回滚

- 风险：客户端对旧返回结构有隐式依赖，迁移后可能出现前端报错。
- 规避：通过统一客户端封装做过渡解析（短期兼容 `ok/success/data`）。
- 回滚：按任务级 commit 拆分，可逐任务回退。
